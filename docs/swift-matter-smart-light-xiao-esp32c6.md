# Swift Matter Smart Light — Build & Hardware Fixes for Seeed XIAO ESP32C6

This document records all changes made to get the [`swift-matter-examples/smart-light`](https://github.com/apple/swift-matter-examples) project building, flashing, and running correctly on a **Seeed Studio XIAO ESP32C6** board. The project was originally published in 2024 and several of its dependencies had drifted since then.

---

## Environment

| Component | Version |
|---|---|
| ESP-IDF | v5.2.1 |
| esp-matter | release/v1.2 |
| Target board | Seeed Studio XIAO ESP32C6 (ESP32-C6FH4) |
| Swift toolchain | Embedded Swift (experimental) |
| Host OS | macOS (Apple Silicon) |
| CMake | 4.x |

## Summary of Problems and Fixes

### 1. Missing `pigweed_environment.gni`

**Problem:** The CHIP/connectedhomeip GN build requires a `pigweed_environment.gni` stub that is normally generated by running `install.sh`. It was absent because that script had never been run.

**Fix:** Created an empty stub at:

```
esp-matter/connectedhomeip/connectedhomeip/build_overrides/pigweed_environment.gni
```

```gni
declare_args() {
}
```

---

### 2. CMake 4.x `ExternalProject_Add` Incompatibility

**Problem:** CMake 4.x changed how `ExternalProject_Add` handles `WORKING_DIRECTORY` when both `SOURCE_DIR` and `BINARY_DIR` are set. It generated a literal `cd "src_path;bin_path"` shell command (semicolon-joined list), which failed immediately.

**File:** `esp-matter/connectedhomeip/connectedhomeip/config/esp32/components/chip/CMakeLists.txt`

**Fix:** Removed `WORKING_DIRECTORY` from the `ExternalProject_Add` call and embedded an explicit `cmake -E chdir` in `BUILD_COMMAND`:

```cmake
# Before
BUILD_COMMAND ninja "esp32"
WORKING_DIRECTORY ${CMAKE_CURRENT_BINARY_DIR}

# After
BUILD_COMMAND ${CMAKE_COMMAND} -E chdir ${CMAKE_CURRENT_BINARY_DIR} ninja "esp32"
```

---

### 3. ZAP Code Generator Not Installed

**Problem:** The CHIP GN build requires the `zap-cli` code generator tool, which is installed via Pigweed's CIPD infrastructure. It was not present on the host.

**Fix:** Manually installed via `cipd`:

```sh
cipd install fuchsia/third_party/zap/mac-amd64 \
  version:2@v2023.10.24-nightly.1 \
  -root ~/.cipd/zap
```

Then symlinked or added to `PATH` so the GN build could locate `zap-cli`.

---

### 4. Swift PCH Could Not Find `riscv/rv_utils.h`

**Problem:** The Swift bridging header pre-compiled header (PCH) build failed because IDF component include directories for the ESP32-C6 target (e.g. `riscv/rv_utils.h`) were not passed to the Swift compiler's `-Xcc -I` flags.

**File:** `swift-matter-examples/smart-light/main/CMakeLists.txt`

**Fix:** Added a loop after the IDF build system is set up to collect all component include directories and append them to `SWIFT_INCLUDES`:

```cmake
idf_build_get_property(idf_component_includes __COMPONENT_INCLUDE_DIRS)
foreach(dir ${idf_component_includes})
    string(CONCAT SWIFT_INCLUDES ${SWIFT_INCLUDES} "-Xcc ")
    string(CONCAT SWIFT_INCLUDES ${SWIFT_INCLUDES} "-I${dir} ")
endforeach()
```

---

### 5. LED Not Responding on Init (`didSet` Not Firing During `init`)

**Problem:** In Swift, `didSet` property observers do not fire for stored properties that are set during `init`. The `LED` class set `brightness` and `enabled` as initial values but never propagated them to the C driver, so the LED started in an unknown state.

**File:** `swift-matter-examples/smart-light/main/LED.swift`

**Fix:** Explicitly called the C driver functions at the end of `LED.init()`:

```swift
init() {
    var config = led_driver_get_config()
    let handle = led_driver_init(&config)
    guard let handle else { fatalError("Failed to initialize handle") }
    self.handle = handle
    // didSet does not fire for stored properties set during init
    led_driver_set_brightness(handle, UInt8(brightness))
    led_driver_set_power(handle, enabled)
}
```

---

### 6. WS2812 Driver Used Deprecated Legacy RMT API

**Problem:** The original `led_driver/ws2812/led_driver.c` used `driver/rmt.h` (legacy RMT API), which was removed in IDF v5.x for the ESP32-C6. Build failed with missing symbols.

**Fix (ws2812 path — superseded by hardware discovery below):**

- Updated `led_driver/idf_component.yml` to use `espressif/led_strip >= 2.0.0` (resolved to v3.0.3).
- Rewrote `ws2812/led_driver.c` to use the new `led_strip` v3 API:

```c
led_strip_config_t strip_config = {
    .strip_gpio_num = config->gpio,
    .max_leds = 1,
    .led_model = LED_MODEL_WS2812,
    .color_component_format = LED_STRIP_COLOR_COMPONENT_FMT_GRB,
    .flags.invert_out = false,
};
led_strip_rmt_config_t rmt_config = {
    .clk_src = RMT_CLK_SRC_DEFAULT,
    .resolution_hz = 10 * 1000 * 1000,
    .flags.with_dma = false,
};
led_strip_handle_t strip = NULL;
esp_err_t err = led_strip_new_rmt_device(&strip_config, &rmt_config, &strip);
```

### 7. Board Has No WS2812 LED (Hardware Discovery)

**Problem:** The XIAO ESP32C6 uses a QFN32 package (ESP32-C6FH4) and has **no WS2812 RGB LED**. The WS2812 driver on GPIO 8 appeared to work (logs showed `led set r:255, g:255, b:255`, GPIO[8] configured, no errors) but nothing was visible because no LED exists on that pin.

The board has two LEDs:
- **Charge LED** (amber, controlled by charge IC — not GPIO controllable)
- **User LED** (GPIO 15, active LOW, single color)

**Fix:** Switched from the `ws2812` LED driver to the `gpio` (LEDC PWM) LED driver.

### 8. Switching to GPIO/LEDC Driver for XIAO User LED

#### 8a. Device CMake configuration

**File:** `esp-matter/device_hal/device/esp32c6_devkit_c/esp_matter_device.cmake`

```cmake
# Before
SET(led_type  ws2812)

# After
SET(led_type  gpio)
```

#### 8b. GPIO pin and channel

**File:** `esp-matter/device_hal/device/esp32c6_devkit_c/device.c`

```c
// Before
#define LED_GPIO_PIN GPIO_NUM_8
#define LED_CHANNEL  LEDC_CHANNEL_0

// After
#define LED_GPIO_PIN GPIO_NUM_15  /* Seeed XIAO ESP32C6 user LED, active LOW */
#define LED_CHANNEL  LEDC_CHANNEL_0
```

Added `#include <driver/ledc.h>` and updated the config initializer accordingly.

#### 8c. LEDC output invert for active-LOW LED

**File:** `esp-matter/device_hal/led_driver/gpio/led_driver.c`

The XIAO user LED is **active LOW** — the GPIO must be driven low to turn it on. The LEDC peripheral supports hardware output inversion via `flags.output_invert`.

```c
ledc_channel_config_t ledc_channel = {
    .gpio_num    = config->gpio,
    .speed_mode  = LEDC_LOW_SPEED_MODE,
    .channel     = config->channel,
    .intr_type   = LEDC_INTR_DISABLE,
    .timer_sel   = LEDC_TIMER_1,
    .duty        = 0,
    .hpoint      = 0,
    .flags.output_invert = 1,   // active-LOW LED
};
```

### 9. RISC-V ABI Crash from Struct Size Change (Two Iterations)

This was the most subtle bug. Two consecutive panics (`Guru Meditation Error`, `Store access fault`, MTVAL=0x64) were caused by attempting to add a field to `led_driver_config_t` to carry the `active_low` flag.

#### Why this matters on RISC-V ILP32

On RISC-V ILP32 (the ABI used by ESP32-C6), small structs returned by value are handled differently depending on their size:

- **≤ 8 bytes**: returned in registers A0 and A1
- **> 8 bytes**: caller allocates stack space and passes a hidden pointer in A0; callee writes through it

`led_driver_config_t` starts as two `int` fields = **8 bytes exactly**, returned in registers.

Swift compiles `led_driver_get_config()` expecting the 8-byte register-return ABI. If the struct grows to 12 bytes (e.g., by adding a `uint8_t` field with padding), the C compiler switches to hidden-pointer ABI, but Swift still reads registers — causing a crash when it tries to write through a garbage pointer.

#### Attempt 1 — Bitfield in anonymous inner struct

```c
// CRASHED — Swift cannot handle anonymous inner struct bitfields
typedef struct {
    int gpio;
    int channel;
    struct { unsigned int active_low: 1; } flags;
} led_driver_config_t;
```

This both crosses the ABI boundary AND uses a C construct (anonymous inner struct with bitfield) that Embedded Swift's C interop cannot represent.

#### Attempt 2 — Plain `uint8_t` field

```c
// CRASHED — struct grew 8 → 12 bytes (alignment padding), crossing ABI boundary
typedef struct {
    int gpio;
    int channel;
    uint8_t active_low;
} led_driver_config_t;
```

Struct layout: `gpio`(4) + `channel`(4) + `active_low`(1) + padding(3) = **12 bytes**. C switches to hidden-pointer return; Swift still used register-return. MTVAL=0x64 (Store access fault writing through a corrupted pointer).

#### Final fix — Keep struct at 8 bytes, hardcode invert in driver

The safest fix that avoids all Swift/C interop issues: keep `led_driver_config_t` at its original 8-byte layout and hardcode `output_invert = 1` directly in the gpio driver, since this driver is only ever used with the XIAO's active-LOW LED.

**`led_driver/include/led_driver.h`** — reverted to original:

```c
typedef struct {
    int gpio;
    int channel;
} led_driver_config_t;
```

**`led_driver/gpio/led_driver.c`** — hardcoded invert:

```c
.flags.output_invert = 1,   // XIAO user LED is active LOW
```

**`device_hal/device/esp32c6_devkit_c/device.c`** — no extra field:

```c
led_driver_config_t config = {
    .gpio    = LED_GPIO_PIN,
    .channel = LED_CHANNEL,
};
```

## Key Lessons

1. **Embedded Swift C interop is strict about struct layout.** Struct size changes that cross the RISC-V ILP32 8-byte register/pointer return boundary will cause silent ABI mismatches and Store access faults at runtime — not at compile time.

2. **Swift `didSet` does not fire during `init`.** Any initial values set as part of stored property initialization must be explicitly propagated to external state (like C drivers).

3. **CMake 4.x changed `ExternalProject_Add` behavior.** `WORKING_DIRECTORY` combined with `SOURCE_DIR`+`BINARY_DIR` is broken; use `cmake -E chdir` in `BUILD_COMMAND` instead.

4. **The Seeed XIAO ESP32C6 (QFN32/FH4 variant) has no WS2812 LED.** The user LED is a plain single-color LED on GPIO 15, active LOW, driven by LEDC PWM.
