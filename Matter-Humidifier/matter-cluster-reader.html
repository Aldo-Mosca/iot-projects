Reference links for Matter cluster & device implementation — Matter-Humidifier project

⭐ For the humidifier specifically: prioritise the mode_select cluster in the esp-matter SDK docs — designed for devices cycling through named modes (Hi / Low / Night / Off). The Fan cluster is also worth a look for its multi-mode semantics.

esp-matter — Official SDK

C++Most Relevant
ESP-Matter Developer Guide
Official Espressif SDK docs. The mode_select cluster section is directly applicable to the humidifier. Also covers custom cluster creation with cluster::create(endpoint, custom_cluster_id, CLUSTER_FLAG_SERVER).
↗ Developer Guide
GitHub Issue #378 — Thermostat Endpoint Pattern
The canonical pattern: create node → build config_t → call endpoint::create() → fetch cluster handle with cluster::get(endpoint, ClusterId) → add optional features. Directly translatable to a humidifier endpoint.
↗ Issue #378
GitHub Issue #1409 — Reading & Writing Cluster Attributes
Demonstrates the attribute::get_val / attribute::update pattern at runtime. Useful for syncing humidifier state back to Apple Home.
↗ Issue #1409

Higher-level wrapper over esp-matter. Working examples for Matter Fan (speed + mode control), Matter Humidity Sensor, and Matter Thermostat. Good for understanding how modes/states map to Matter attributes before dropping to raw esp-matter API calls.

matter.js — Full Matter Protocol in TypeScript
Complete TypeScript implementation of the Matter spec. Strong typing across all clusters — the best place to browse what attributes and commands a cluster actually exposes before writing C++ or Swift.

rs-matter

RustStructural Reference
rs-matter Cluster Implementation Patterns
Detailed documentation of cluster implementation patterns — from simple data-driven to complex coupled clusters (e.g. LevelControl + OnOff). The Fan cluster example shows how to implement hooks, configure attributes, and chain the handler onto an endpoint. Maps well to the Swift shim architecture.

Implementing New Clusters & Device Types
Most spec-correct reference. Covers the full flow: ember attribute storage, PluginServerCallback, AttributeAccessInterfaceRegistry overrides, and bypassing ember storage. Use this to know what's mandatory vs. optional per the spec.
